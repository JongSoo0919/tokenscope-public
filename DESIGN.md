## Problem Statement

Claude Code Pro는 토큰 한도가 정해져 있고, 예산을 늘릴 수 없다.  
사용률은 어렴풋이 알지만, 어디서 토큰이 낭비되는지 모른다.  
ccusage 같은 도구는 "얼마나 썼나"는 보여주지만, "왜 낭비되는가"와 "어떻게 고치는가"는 없다.

같은 예산으로 더 많은 작업을 돌리고 싶다 — 절약이 아니라 **같은 돈으로 더 강해지기**.

---

## What Makes This Cool

**의사처럼 진단하고 처방전을 쓰는 도구.**

기존 도구: 숫자 보여줌 → 사용자가 알아서 고침  
이 도구: 패턴 분류 → 수치화된 처방 → 버튼 하나로 파일 수정 → 다음 세션에서 효과 확인

낭비 패턴 분류 taxonomy (v1: 3개 구현, v2에서 확장):
- `CONTEXT_BLOAT` — system role 전체 토큰이 세션 총 토큰의 30% 이상 (v1)
- `RETRY_STORM` — 동일 user 메시지(해시 기준)가 3회 이상 반복 (v1)
- `TOOL_THRASH` — 동일 tool name이 연속 3회 이상 + 모든 tool_result가 error (v1)
- `HALLUCINATION_RECOVERY` — 모델이 자기 이전 응답을 부정하는 패턴 (v2 예정)

이 분류 자체가 이 프로젝트의 핵심 IP다. 막연한 "낭비 찾기"가 아니라 진단 가능한 범주.

## Constraints

- 로컬 파일만 사용 — 네트워크, LLM 호출 없음. 오프라인 완전 작동
- Claude Code Pro 구독자 대상 — `~/.claude/projects/` JSONL 포맷 기준
- macOS 우선 (추후 Linux/Windows)
- 규칙 기반 분석 — LLM 없이도 충분한 패턴 감지

## Premises

1. ✅ 핵심 가치는 대시보드(통계)가 아니라 **진단 + 수정안 생성** — ccusage가 이미 통계는 해결
2. ✅ Claude Code 먼저, Codex/Gemini CLI 멀티 에이전트 지원은 v2
3. ✅ 데스크탑 앱(Tauri) — 웹은 로컬 파일 접근에 한계

## Cross-Model Perspective

Claude subagent cold read:

**가장 쿨한 버전** — 지속적 최적화 루프. 세션 끝날 때마다 자동으로 JSONL 분석 → diff 보여주며 "이 섹션 지우면 세션당 2,100토큰 절약" 수치 부착 → 적용 버튼 하나 → 다음 세션 후 "-31% 확인됨" 피드백.

**핵심 인용** — "금액을 늘릴 수 없으니 소모되는 금액을 줄이고싶어" — 절약이 아니라 같은 돈으로 더 강해지기가 감정적 코어.

**50% 해결 오픈소스** — ccusage: JSONL 파싱, 토큰 집계, 비용 계산 이미 구현. 만들 것: 패턴 분석 엔진 + 분류기 + 처방 생성기 + 파일 수정 액션.

**빌드 순서** — Tauri + React, 규칙 기반(LLM 없음, 오프라인). 1단계=파이프라인(JSONL → 패턴 JSON, CLI로 먼저 검증), 2단계=UI + 파일 수정 버튼 + 백업/되돌리기. Effort: M (1-2주).

## Approaches Considered

### Approach A: CLI 도구 (최소 viable)
- Summary: Node.js/TS CLI. ccusage 파싱 + 패턴 분석 + 개선안 텍스트 출력. 사용자가 직접 적용.
- Effort: S (3-5일) | Risk: Low
- Pros: 빠른 핵심 검증, UI 없이 분석 엔진 먼저 테스트
- Cons: 파일 수정 액션 없음, 적용 마찰 높음, UX 완성 안 됨
- Completeness: 5/10

### Approach B: Tauri 데스크탑 앱 (선택)
- Summary: Tauri + React + TypeScript. 세션 목록 + 패턴 감지 + one-click CLAUDE.md 수정 + 전후 비교.
- Effort: M (1-2주) | Risk: Low
- Pros: 진짜 "적용" UX, 파일 수정 Rust로 안전하게, 멀티 에이전트 확장 구조
- Cons: Tauri 러닝커브 (처음이면)
- Reuses: ccusage JSONL 파싱 로직 참고
- Completeness: 9/10

### Approach C: macOS 메뉴바 앱 (창의적)
- Summary: 백그라운드 상주 + file watcher. 세션 끝나면 자동 분석. 메뉴바에 "오늘 2,100토큰 절약됨" 표시.
- Effort: L (2-3주) | Risk: Med (daemon + file watcher 복잡도)
- Pros: "그냥 돌아가는" 경험, 가장 cool
- Cons: 분석 엔진 검증 전에 daemon 복잡도까지 동시 디버깅 위험
- Completeness: 8/10

## Recommended Approach — Approach B 상세 설계

### 기술 스택

| 레이어 | 도구 | 이유 |
|--------|------|------|
| 셸 | Tauri v2 (Rust + TypeScript) | 로컬 파일 접근 Rust로 안전, 번들 크기 작음 |
| UI | React + TypeScript | 컴포넌트 구조, 타입 안전성 |
| 스타일 | Tailwind CSS + shadcn/ui | 빠른 스타일링, 기본 컴포넌트 |
| 파싱 | 직접 구현 (ccusage 로직 참고) | 의존성 최소화 |
| 분석 | 순수 규칙 기반 TypeScript | 오프라인, LLM 불필요 |
| 빌드 | yarn + Vite | 기존 환경 일치 |

### 데이터 파이프라인

```
~/.claude/projects/
  └── {project-hash}/
      └── *.jsonl   ← 각 대화 세션 (1줄 = 1 message)

JSONL 실제 구조 (role별):
// system message
{ "role": "system", "content": "...", "usage": { "input_tokens": N } }

// user message
{ "role": "user", "content": "...", "usage": { "input_tokens": N } }

// assistant message
{ "role": "assistant", "content": "...",
  "usage": { "input_tokens": N, "output_tokens": N, "cache_read_input_tokens": N } }

// tool_use (assistant가 tool 호출)
{ "role": "assistant", "content": [{ "type": "tool_use", "id": "...", "name": "Bash", "input": {...} }],
  "usage": { "input_tokens": N, "output_tokens": N } }

// tool_result (tool 응답)
{ "role": "user", "content": [{ "type": "tool_result", "tool_use_id": "...",
  "content": "...", "is_error": false }],
  "usage": { "input_tokens": N } }
```

**스키마 안정성 전략**: JSONL 포맷은 Anthropic 내부 구현으로 공식 API 보장 없음.
- 파서 어댑터 패턴 적용: `SchemaV1Parser` → `SessionData` (내부 표준 구조)
- 파싱 시 `role` / `usage` 필드 존재 검증. 없으면 해당 메시지 skip + 경고 로그
- 포맷 변경 감지: 파싱 성공률이 50% 미만이면 "로그 포맷 변경 감지됨" 알림

**대용량 처리 전략**: 수개월 사용 시 JSONL이 수백 MB 가능.
- 파싱 결과를 `~/.tokenscope/cache/{session-hash}.json`에 캐시
- 파일 mtime 비교로 변경된 세션만 재파싱 (증분 파싱)
- 앱 시작 시 최근 30일치만 자동 로드, 과거 데이터는 수동 요청

**토큰 비용 계산**: JSONL `usage` 필드에 모델 정보가 없으면 세션 파일명/메타에서 추론.
기본 단가 (2026-04 기준, 앱 내 설정에서 수정 가능):

| 모델 | input (per 1M) | output (per 1M) | cache_read (per 1M) |
|------|---------------|-----------------|---------------------|
| claude-sonnet-4 | $3.00 | $15.00 | $0.30 |
| claude-opus-4 | $15.00 | $75.00 | $1.50 |
| claude-haiku-4 | $0.80 | $4.00 | $0.08 |

**Rust/TS 책임 경계**:
- `read_logs.rs` (Rust Tauri command): 파일 I/O만 담당 — `~/.claude/projects/` 디렉토리 탐색, 파일 목록 반환, raw JSONL 텍스트 반환
- `parser.ts` (TypeScript): JSONL 파싱 + 구조화 담당 — raw 텍스트 → `SessionData[]`

**에러 처리 매트릭스**:

| 에러 상황 | 원인 | UI 표현 |
|----------|------|---------|
| 경로 없음 | Claude Code 미설치 또는 경로 다름 | "~/.claude/projects/ 를 찾을 수 없습니다. Claude Code가 설치되어 있나요?" + 경로 수동 입력 옵션 |
| 권한 부족 | macOS 파일 접근 차단 | "파일 접근 권한이 없습니다. 시스템 설정 > 개인정보 보호 > 파일 접근을 확인해주세요." |
| 파일 잠김 | 세션 진행 중 | 해당 파일 skip + "현재 진행 중인 세션은 종료 후 분석됩니다" 인라인 표시 |
| 포맷 불일치 | JSONL 스키마 변경 | "로그 포맷이 변경된 것 같습니다. 업데이트를 확인해주세요." + GitHub link |

파싱 단계:
1. `read_logs.rs`로 파일 목록 + raw 텍스트 수신
2. `parser.ts`의 `SchemaV1Parser`로 JSONL 파싱 (위 에러 매트릭스 적용)
3. `usage` 필드 집계 → 세션 총 토큰 계산
4. 패턴 분석 엔진에 메시지 배열 전달
5. 세션 메타데이터 생성 (총 토큰, 비용 추정, 날짜, 패턴 목록)

### 패턴 분류 엔진 (규칙 기반, v1: 3개 패턴)

```typescript
// v1 스코프: 3개 패턴만 구현. HALLUCINATION_RECOVERY는 v2.
type WastePattern =
  | { type: 'CONTEXT_BLOAT'; systemTokenRatio: number; systemTokens: number; totalTokens: number }
  | { type: 'RETRY_STORM'; count: number; repeatedPromptHash: string; repeatedPromptPreview: string }
  | { type: 'TOOL_THRASH'; toolName: string; consecutiveErrorCalls: number }

// 판별 heuristic
// CONTEXT_BLOAT:
//   systemTokens (role==="system" 메시지의 usage.input_tokens 합계) / sessionTotalTokens > 0.30
//   참고: system role 토큰 = CLAUDE.md + tool schema 포함. CLAUDE.md만 분리 불가.
//   대신 CLAUDE.md 파일 내용을 직접 읽어 섹션별 토큰 근사치 계산 (tiktoken-wasm 사용)

// RETRY_STORM:
//   대상: 세션 내에서 role==="user" 이고 content가 string 타입인 메시지만
//   (tool_result처럼 content가 배열인 메시지는 제외)
//   동일 MD5 해시가 세션 내 3회 이상 등장 → RETRY_STORM

// TOOL_THRASH:
//   메시지 배열에서 role==="assistant" 이고 content[].type==="tool_use"인 것만 순서대로 추출
//   추출된 리스트에서 동일 name이 연속 3회 이상 등장
//   각각에 매칭되는 tool_result (tool_use_id로 매칭)가 모두 error인 경우
//   "error" 판별: is_error === true, 또는 content 문자열에 /^Error:/i 포함
```

**토큰 계산 방법**:
- 세션 총 토큰: JSONL `usage` 필드 값 사용 (정확)
- CLAUDE.md 섹션별 예측: CLAUDE.md 원본을 `tiktoken-wasm`(cl100k_base)으로 섹션(## 기준) 분할 후 각 섹션 토큰 독립 추정. 이는 system role 토큰에서 분리하는 것이 아닌 "이 섹션을 지우면 약 N토큰 줄어들 것"이라는 근사치. UI에 "~ 예상"으로 표시.
- `tiktoken-wasm` 번들 크기가 10MB 초과 시: fallback으로 `/3` 일률 적용 (한영 혼합 기준 보수적 추정). 언어별 분기 없음.

### 처방 생성기

각 패턴마다 하드코딩된 처방 + 적용 액션:

```
CONTEXT_BLOAT:
  진단: "system role이 세션 토큰의 {ratio}% — CLAUDE.md + tool schema 합계 {N}토큰"
  처방: CLAUDE.md를 섹션(## 기준)으로 분할 → 각 섹션 토큰 계산
        → "## 전역 지침: ~340토큰. 이 섹션 제거 시 세션당 약 340토큰 절약 예상"
  적용: 선택한 섹션을 CLAUDE.md에서 삭제 (수정 전 자동 백업)
  표시: before/after diff (DiffViewer 컴포넌트)

RETRY_STORM:
  진단: "동일 프롬프트 {count}회 반복 감지 — 에러 루프 가능성"
        (반복된 프롬프트 앞 50자 미리보기)
  처방: CLAUDE.md에 가드레일 텍스트 삽입 제안:
        "## 재시도 가드레일\n- 동일 명령이 2회 이상 실패 시 중단하고 사용자에게 보고"
  적용: CLAUDE.md 맨 끝에 섹션 추가 (수정 전 자동 백업)

TOOL_THRASH:
  진단: "{toolName} tool {count}회 연속 실패"
  처방: 해당 세션의 실패 내용 요약 표시 (참고용)
        → 이 패턴은 자동 파일 수정 없음 (v1). 사용자가 수동으로 skill 수정하도록 안내.
        → "이 패턴이 반복되면 skills/{domain}/SKILL.md에 에러 처리 절차를 추가하세요"
```

**파일 수정 안전장치**:
- 수정 전 원본을 `~/.tokenscope/backups/{filename}-{timestamp}.bak`에 자동 저장
- UI에 "되돌리기" 버튼 노출 (최근 10개 백업 보존)
- `apply_fix.rs`는 백업 성공 확인 후에만 원본 수정

### 핵심 흐름

## Risk Management

**JSONL 포맷 안정성** (존재론적 리스크):
- Anthropic 내부 구현이므로 공식 보장 없음. 경로 변경, 스키마 변경, 암호화 가능성 있음.
- 대응: (1) 파서를 어댑터 패턴으로 교체 용이하게, (2) 포맷 변경 감지 시 앱 내 알림, (3) Anthropic 공식 API 제공 시 즉시 마이그레이션
- 구현 시작 전: 실제 `~/.claude/projects/` 내 JSONL 3-5개 파일 파싱해서 스키마 확정 필수

**Tauri v2 fs 접근 범위** (`tauri.conf.json` 필수 설정):
```json
{
  "plugins": {
    "fs": {
      "scope": ["$HOME/.claude/**", "$HOME/.tokenscope/**"],
      "all": false
    }
  }
}
```
Tauri v2는 v1보다 보안 모델이 엄격함. 위 scope 없이는 홈 디렉토리 접근 차단됨.
공식 문서: https://v2.tauri.app/plugin/file-system/

**"수술 효과" 비교 로직**:
수정 시점 timestamp 기록 → 해당 프로젝트의 수정 전 최근 5세션 평균 system 토큰 비율 vs 수정 후 최근 5세션 평균 비교. 세션 수가 5 미만이면 있는 만큼만 사용. 결과: "수정 전 평균 38% → 수정 후 평균 22% (절약: 세션당 ~1,800토큰)".

## Open Questions

1. **캐시 토큰 처리**: `cache_read_input_tokens`는 일반 input 단가의 10%로 환산 (결정).
2. **JSONL에 모델 정보 포함 여부**: 실제 JSONL 파일에 모델명 필드가 있는지 확인 필요. 없으면 사용자가 설정에서 기본 모델 선택.
3. **tiktoken-wasm 번들 크기**: 앱 번들에 포함 시 크기 확인 필요. 10MB 초과 시 char-based 근사치로 fallback 고려.
4. **v2 에이전트 로그 경로** (v2 구현 전 확인):
   - Codex CLI: `~/.codex/` 또는 `~/.openai-codex/` (미확인)
   - Gemini CLI: `~/.gemini/` (미확인)
5. **HALLUCINATION_RECOVERY**: v2 예정. 규칙 기반 감지 시 false positive 높아 신뢰도 확보가 선행되어야 함.

## What I noticed about how you think

- "금액을 늘릴 수 없으니" — 제약을 전제로 받아들이고, 그 안에서 최대치를 찾는 방식이다. 불가능한 걸 바꾸려 하지 않고, 가능한 걸 최적화한다. 엔지니어의 본능.
- "세개를 한번에 묶는건 불가능한가?" — 기능을 분리해서 생각하지 않는다. 사용자 입장에서 하나의 여정으로 본다. 이 관점이 좋은 UX를 만든다.
- "나" — 첫 유저를 본인으로 잡은 것이 이 프로젝트의 가장 강한 신호다. 리서치가 필요 없다. 매일 직접 쓰면서 뭐가 불편한지 바로 알 수 있다.
- 하네스 엔지니어링 배경이 있어서, 이 도구의 자연스러운 확장이 "harness 자동 생성"이라는 걸 이미 알고 있었다. 직접 경험에서 나온 통찰.
