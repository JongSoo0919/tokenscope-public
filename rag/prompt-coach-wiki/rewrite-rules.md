# TokenScope 질문 재작성 규칙

## 입력 판단 순서

1. 사용자의 실제 의도를 한 문장으로 요약한다.
2. 질문이 기획, 구현, 디버깅, 리뷰, 검증 중 어디에 가까운지 분류한다.
3. 낭비 가능성을 찾는다.
4. 다음 질문을 더 작고 검증 가능하게 다시 쓴다.

## 낭비 가능성 판정

- 범위 없음: 대상 파일, 프로젝트, 화면, API가 없다.
- 완료 조건 없음: 언제 끝나는지 알 수 없다.
- 단계 혼합: 기획과 구현, 구현과 검증이 한 번에 들어 있다.
- 금지 조건 없음: 삭제, 전체 재작성, 대규모 리팩터링 위험이 있다.
- 에러 정보 부족: 실패 로그, 재현 명령, 기대 결과가 없다.
- 이전 맥락 과다: 이전 대화 전체를 다시 들고 오려 한다.

## 출력 형식

TokenScope에 표시할 답변은 항상 아래 형식을 따른다.

```text
요약:
<현재 질문의 의도 한 문장>

문제:
- <토큰 낭비 또는 실패 위험>
- <작업 범위 위험>

다음에는 이렇게 질문하세요:
<개선된 질문 템플릿>

토큰 절약 포인트:
- <왜 이 질문이 더 싼지>
- <어떤 탐색/반복을 줄이는지>
```

## 재작성 기준

- 사용자의 말투를 완전히 바꾸지 말고 목적을 보존한다.
- 너무 긴 만능 프롬프트로 만들지 않는다.
- 한 번의 질문에는 하나의 목표만 둔다.
- 질문이 이미 좋다면 불필요하게 고치지 말고 유지할 점을 알려준다.
- 불확실한 정보는 추측하지 말고 `<여기에 파일 경로>`, `<에러 로그 30줄>`처럼 빈칸으로 둔다.
